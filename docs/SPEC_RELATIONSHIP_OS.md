# Welian 关系操作系统产品规约

> **v2.0 核心跃迁：从"关系管理工具"到"关系操作系统"。**
>
> v1 定义了双关系模型和四个核心场景（记/问/拟/报），验证了产品价值。
> v2 把 Welian 从一个"帮你管关系的工具"升维为一个**关系数据的操作系统**——
> 不仅存储和检索关系数据，而是成为所有关系相关场景的**智能中间层**：
> 日历、信号、图谱、跨平台同步、第三方集成，全部由 Welian 统一调度。
>
> 类比：v1 是"通讯录 2.0"，v2 是"关系的 iOS"——上面可以跑无数关系类应用，
> 但底层数据模型、伦理边界、智能引擎由 Welian 统一提供。

---

## §0. 为什么要从"工具"升级为"操作系统"

### 0.1 v1 验证了什么

| 已验证 | 证据 |
|:----|:----|
| 双关系模型有效 | 撬动/维系二分解决了"对家人做ROI"的伦理困境，用户接受度高 |
| 对话是最佳交互 | 记/问/拟/报四个动词覆盖了关系维护的核心场景 |
| 数据飞轮是核心瓶颈 | timeline 34条/月，功能跑在数据前面 |
| 主动推送是留存关键 | 周报是用户回来的主要理由 |
| 本地数据主权是差异化 | Clay/Dex 依赖 Gmail+LinkedIn 在中国不可用 |

### 0.2 v1 的天花板

| 天花板 | 表现 | 根因 |
|:----|:----|:----|
| 工具心智 | 用户认为"用完就走"，不是"常驻后台" | 产品只在用户主动打开时才有价值 |
| 数据孤岛 | 关系数据锁在 Welian 内部，不与用户的其他工具互通 | 缺乏输出通道（日历/通知/第三方） |
| 信息单向 | 只消费用户输入的数据，不主动从外部世界获取信息 | 缺乏信号输入层 |
| 单点价值 | 每个功能独立，1+1=1.5 而非 1+1>2 | 缺乏跨功能的编排能力 |

### 0.3 操作系统隐喻

| 层 | iOS 等价物 | Welian 等价物 |
|:----|:----|:----|
| **内核** | Darwin（进程/内存/文件系统） | 双关系引擎（撬动/维系/双重 + 数据模型） |
| **系统服务** | CoreLocation/CoreData/CloudKit | 关系图谱/信号引擎/日历同步/待办调度 |
| **框架** | UIKit/SwiftUI/HealthKit | 关系API（记/问/拟/报 + 日历 + 信号 + 图谱） |
| **应用层** | App Store 上的 App | 场景应用（销售助手/家庭关怀/人脉分析/年度报告） |
| **生态** | 开发者 + 第三方集成 | 开源 CLI + MCP Server + 第三方关系类工具 |

**关键洞察**：v1 做的是"应用层"的事（记/问/拟/报），v2 要把底层抽出来成为"系统服务"，
让上层场景应用可以像调用系统 API 一样调用关系能力。

│                    场景应用层                              │

│  （每个场景调用下层 API 组合出完整体验）                     │
├─────────────────────────────────────────────────────────┤
│                    关系框架层                              │
│  记 · 问 · 拟 · 报 · 日历 · 信号 · 图谱 · 待办 · 分享       │
│  （九个关系原语，可被任何场景应用调用）                      │
├─────────────────────────────────────────────────────────┤
│                    系统服务层（v2 新增）                    │
│  关系图谱引擎 · 信号聚合引擎 · 日历同步服务                  │
│  待办调度器 · 通知中心 · 数据同步总线                       │
│  （独立于 UI 的后台服务，持续运行）                         │
├─────────────────────────────────────────────────────────┤
│                    内核层（继承 v1）                        │
│  双关系引擎 · 数据模型 · LLM Gateway · 隐私架构             │
│  （Welian 的地基，v1 已验证）                              │
└─────────────────────────────────────────────────────────┘

### 3.1 内核层（继承 v1，稳定）

| 模块 | 说明 | 状态 |
|:----|:----|:----|
| 双关系引擎 | 撬动/维系/双重分类 + 伦理边界 | ✅ v1 已验证 |
| 数据模型 | contacts.json + timeline + todos + memories | ✅ v1 已验证 |
| LLM Gateway | 多模型路由（MiniMax/DeepSeek/Qwen/GLM） | ✅ 已上线 |
| 隐私架构 | 端云分离 + 最小上下文 + 即用即焚 | ✅ 已上线 |

### 3.2 系统服务层（v2 新增，核心投资）

这是 v1 到 v2 的本质区别——四个独立后台服务，让 Welian 从"打开才有用"变成"常驻后台持续运转"。

#### 3.2.1 日历同步服务

**定位**：关系数据 → 时间维度的输出通道

| 能力 | 说明 | 状态 |
|:----|:----|:----|
| macOS 日历同步 | AppleScript 本地同步，launchd 定时任务 | ✅ 已完成（calendar_sync.py） |
| .ics 订阅链接 | 生成订阅 URL，导入 Google/Apple/Outlook | 🔒 待开发 |
| Google Calendar API | OAuth 授权，双向同步 | 🔒 待开发 |
| 智能日程编排 | AI 根据关系优先级和用户日程推荐最佳联系时间 | 🔮 未来 |

**设计原则**：
- 待办自动同步，不需要用户手动录入日历
- 维系型关系的提醒（生日/纪念日）优先级高于撬动型待办
- 日历事件包含关系上下文（上次聊了什么、该聊什么）

#### 3.2.2 信号聚合引擎

**定位**：外部世界 → 关系网络的智能过滤器

| 能力 | 说明 | 状态 |
|:----|:----|:----|
| HN 信号 | 每日抓取 HN 热门，结合关系网络个性化筛选 | ✅ 已上线 |
| 信号→关系映射 | "这条新闻和你的哪个联系人相关，该转发给谁" | ✅ 已上线 |
| 多源扩展 | 微博/即刻/小红书/Twitter 等中文和全球信号源 | 🔒 待开发 |
| 信号→行动闭环 | 信号 → 推荐行动 → 一键转发/记录/设待办 | 🔒 待开发 |
| 用户自定义信号源 | "关注 AI Agent 领域" → 定向抓取相关内容 | 🔮 未来 |

**设计原则**：
- 信号不是"今日热点"，是"和你关系网络相关的信息"
- 每条信号必须关联到至少一个联系人或一个待办，否则不推送
- 信号是关系经营的契机，不是信息消费的入口

#### 3.2.3 关系图谱引擎

**定位**：从线性列表到网络结构的升维

| 能力 | 说明 | 状态 |
|:----|:----|:----|
| 图谱数据模型 | 联系人为节点，关系为边（共同领域/引荐路径/互动频率） | 🔒 待开发 |
| 可视化 | 力导向图展示社交网络结构，发现忽略的角落 | 🔒 待开发 |
| 引荐路径 | "你和张总之间有 2 度分隔，通过老李可以搭桥" | 🔮 未来（数据达标后解冻） |
| 关系健康度 | AI 定期评估网络健康度（哪些在冷却/升温/休眠） | 🔒 待开发 |
| 社群发现 | 自动识别你的社交圈层（同事圈/校友圈/行业圈） | 🔮 未来 |

**设计原则**：
- 图谱是观察工具不是评判工具——展示结构，不给人贴标签
- 引荐路径只对撬动型关系计算，维系型关系不参与"利用"逻辑
- 健康度只用于撬动型关系，维系型关系的"健康"不可量化

#### 3.2.4 待办调度器

**定位**：从静态待办列表到智能任务编排

| 能力 | 说明 | 状态 |
|:----|:----|:----|
| AI 待办提取 | 从对话中自动提取待办，关联联系人 | ✅ 已上线 |
| 优先级排序 | 按关系重要性 + 截止日期 + 冷却状态排序 | ✅ 已上线 |
| 日历推送 | 待办自动同步到日历 | ✅ 本地版 / 🔒 云端版待开发 |
| 智能提醒 | "这个待办已经拖了 3 天，对方是重要客户，建议今天处理" | 🔒 待开发 |
| 兑现追踪 | 待办完成后自动关联到关系兑现记录 | 🔒 待开发 |

### 3.3 关系框架层（九个原语）

v1 的四个动词（记/问/拟/报）+ v2 新增五个原语（日历/信号/图谱/待办/分享），构成完整的关系操作 API。

| 原语 | 输入 | 输出 | v1/v2 |
|:----|:----|:----|:----:|
| **记** | 自然语言 | 结构化联系人+互动+待办 | v1 |
| **问** | 联系人+场景 | 速览（上次话题/待办/近况） | v1 |
| **拟** | 场景+对象 | 可发送的消息草稿 | v1 |
| **报** | 时间范围 | 周报/月报/年度报告 | v1 |
| **📅 日历** | 待办+关系上下文 | 日历事件+提醒 | v2 |
| **📡 信号** | 外部信息源+关系网络 | 个性化信号简报+行动建议 | v2 |
| **🕸 图谱** | 全量联系人+互动数据 | 关系网络可视化+健康度 | v2 |
| **✅ 待办** | 对话/手动输入 | 排序后的待办队列+日历推送 | v2 |
| **📤 分享** | 报告/信号/战报 | 可分享的卡片/图片/PDF | v2 |

### 3.4 场景应用层

场景应用是关系原语的具体组合，面向特定用户群体的完整体验。

| 场景应用 | 目标用户 | 原语组合 | 状态 |
|:----|:----|:----|:----|
| **销售助手** | 销售/BD/创业者 | 记+问+拟+日历+待办+图谱 | 🔒 待开发 |
| **家庭关怀** | 异地/海外/新手父母 | 记+报+日历（生日/纪念日）+信号 | 🔒 待开发 |
| **人脉分析** | 高管/投资人 | 图谱+报+信号+待办 | 🔒 待开发 |
| **节日管家** | 所有人 | 拟+日历+信号（节日相关）+分享 | 🔒 待开发 |
| **年度报告** | 所有人 | 报+图谱+分享 | 🔒 待开发 |
| **角色扮演** | 新用户（获客前厅） | 记+问+拟 | ✅ 已上线 |

## §7. 数据与隐私架构（继承 v1 + 扩展）

### 7.1 端云分离（继承 v1）

> **数据归你，智能来云。**（v1 §7.1 不变）

### 7.2 v2 新增：数据输出权

v1 的隐私架构强调"数据不出去"，v2 补充"数据可以出去——但只有用户能决定"：

| 输出通道 | 说明 | 用户控制 |
|:----|:----|:----|
| 日历同步 | 待办→日历事件 | 用户选择哪些待办同步 |
| .ics 订阅 | 生成只读订阅链接 | 用户可随时撤销链接 |
| 数据导出 | JSON/CSV 全量导出 | 永久免费，不锁定 |
| MCP Server | 第三方 AI 工具调用 | 用户授权制，可随时撤销 |
| API 访问 | 个人 API Key | 🔮 未来 |

### 7.3 合规（继承 v1）

中国法规应对不变（PIPL/生成式AI/微信生态），v2 新增：

| 法规 | v2 新增应对 |
|:----|:----|
| 日历权限 | 日历同步需明确授权，同步数据仅含待办摘要，不含完整对话 |
| 第三方集成 | MCP Server/API 访问需 OAuth 授权，第三方不获得原始数据 |

## §11. 不做什么（继承 v1 十不做 + v2 新增三不做）

**继承 v1 十不做：**
1. 不做社交平台
2. 不做聊天记录自动导入
3. 不对维系型关系做 ROI/排序/冷却
4. 不做幸福评分
5. 不做广告
6. 不做关系数据变现
7. 不做自动发消息/自动点赞
8. 不做无限建议
9. 不做订阅锁数据
10. 不做"附近的人"/LBS 社交

**v2 新增：**
11. **不做关系数据交易平台**——不卖关系图谱，不卖社交网络分析，不卖"人脉画像"
12. **不做自动社交机器人**——不自动加好友/自动聊天/自动维护关系，AI 辅助但不替代人的用心
13. **不做关系评分排名**——不对关系做"价值排名"或"重要性评分"暴露给用户，图谱用于观察不用于评判

## §12. 竞争格局（v2 新增）

| 竞品 | 定位 | Welian 差异 |
|:----|:----|:----|
| Clay | AI CRM（美国） | 双关系模型 + 中国可用 + 隐私优先 |
| Dex | Personal CRM（美国） | 主动推送 + 信号引擎 + 日历同步 |
| Notion Contacts | 通用工具的联系人模板 | 垂直 AI 引擎 + 双关系模型 + 场景应用 |
| 通讯录/名片夹 | 存储 | AI 理解 + 主动提醒 + 关系图谱 |
| 微信备注 | 最简关系记录 | 结构化 + AI 增强 + 跨平台 + 日历 |

**Welian 的护城河**：
1. **双关系模型**——竞品都是单一模型，对家人做 ROI 是伦理硬伤
2. **主动推送矩阵**——竞品都是"打开才有用"，Welian 是"常驻后台"
3. **信号引擎**——竞品只消费用户输入，Welian 主动从外部世界获取信息
4. **中国可用**——Clay/Dex 依赖 Gmail+LinkedIn，中国不可用
5. **开源信任**——CLI 开源 + 数据可审计 + 随时导出
